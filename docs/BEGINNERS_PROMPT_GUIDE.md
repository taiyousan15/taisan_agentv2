## 1. 基本の考え方

### 覚えておくべき3つのルール

```
┌─────────────────────────────────────────────────────────────┐
│  ルール1: 普通の日本語でOK                                   │
│  ルール2: 「〜して」で終わればOK                             │
│  ルール3: 詳しく言うほど精度が上がる                         │
└─────────────────────────────────────────────────────────────┘
```

### 良い例 vs 悪い例

| ❌ 悪い例（難しく考えすぎ） | ✅ 良い例（シンプル） |
|---------------------------|---------------------|
| 「Task toolでsubagent_type=bug-fixerを...」 | 「このバグを直して」 |
| 「/sales-letter --product "..."を実行」 | 「セールスレターを書いて」 |
| 「dependency-validatorエージェントで検証」 | 「依存関係をチェックして」 |

**ポイント**: Claudeが自動的に最適なエージェントやスキルを選びます。

「なぜ動かないのか調べて」
「エラーログを分析して原因を特定して」

### エラーの伝え方のコツ

```
# 最も効果的な伝え方
「npm run build したらこんなエラーが出ました:

error TS2339: Property 'foo' does not exist on type 'Bar'.
  src/services/user.service.ts:42:15

修正してください」

# これだけでClaudeは自動的に:
# 1. エラーの種類を特定
# 2. 該当ファイルを読む
# 3. 修正案を提示
# 4. あなたの承認を得て修正
```

## 10. MCP（外部サービス連携）

### MCPとは？

```
MCP = 外部サービスと連携する仕組み

例:
- GitHub → Issue作成、PR操作
- Slack → メッセージ送信
- データベース → クエリ実行
- Docker → コンテナ操作
```

### MCPを意識する必要はない

```
# 難しく考えなくてOK
# 普通に依頼すれば、Claudeが自動でMCPを使います

❌ 「github MCPを使ってIssueを作成して」
✅ 「GitHubにIssueを作って」

❌ 「postgres MCPでSELECTクエリを実行」
✅ 「ユーザー数を数えて」
```

### 具体的なフレーズ例

```
# GitHub
「Issueを作って」
「PRをマージして」
「PRにコメントして」

# データベース
「ユーザー数を教えて」
「最近登録したユーザーを見せて」
「このクエリを実行して」

# Docker
「コンテナを起動して」
「ログを見せて」
「コンテナを再起動して」

# Notion（設定されている場合）
「Notionに議事録を作って」
「ナレッジベースを検索して」

# Slack（設定されている場合）
「Slackに通知して」
「#generalに投稿して」
```

### MCPの状態確認

```
「MCPの状態を確認して」
「利用可能なMCPを教えて」
「GitHubに接続できてる？」
```
